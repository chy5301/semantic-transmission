## 1. 背景与目标（目标不变）

负责人下达硬任务：**6 月 30 日前**交付一个**输入视频流、输出生成视频流**的程序（CUDA 环境，不要求国产化），目标是尽可能替代无人车远程遥操作的视频画面（延迟/清晰度/一致性都要尽量接近真实遥控）。合同基础功能是保底，准实时遥控替代是真实目标。

本文档把已评估的 9 天技术方案**压缩为 6 天冲刺 + 短期/中期/最终三层规划**，并据此更新 `docs/ROADMAP.md`，作为后续每日 brainstorm / spec / plan 的依据。**本轮不产出 writing-plans 级实施计划**——每日任务开始时基于前一日 PoC 结果重新 brainstorm 当天 spec（见 §8）。

## 2. 压缩策略（9 天 → 6 天，激进塞目标版）

四个压缩手法：

1. **保底版 5 天 → 3 天**（D1–D3 紧凑做完闭环 + relay + 评估 + 真实视频测试），第一周 PoC 全程并行；
2. **M1 保底版达成从 6/26 提前到 6/24**，给目标版腾出 D4–D6 三天；
3. **目标版 PoC 4 天 → 3 天**（关键帧主线 + 插帧 + 超分 + 流式雏形）；
4. **砍出 6 天窗口**：Qwen3-VL-4B 升级整体推到 7 月（不阻塞、降风险）；深度图保底版用单帧 Depth-Anything-V2，视频版（Video-Depth-Anything）作目标版 PoC。

## 3. 短期：6 天冲刺编排（6/22–6/27）

**关键并行原则**：所有 PoC（H1/H2/H3）、权重下载、TensorRT 引擎编译都与保底闭环**物理解耦**，可在独立分支/环境并行，不占用 M1 关键路径。单卡 RTX 5090 的 GPU 争用通过小分辨率/错峰缓解。

## 4. 中期规划（7 月）

目标版工程化，逐项把 D4–D6 的 PoC 提升到可用：

- 主线模型（klein 或 Qwen-Edit）参数优化与质量打磨；
- 流式 I/O 从雏形到生产化（相机/RTSP 输入、生成帧编码回流、背压/缓冲）；
- 深度图升级 Video-Depth-Anything（时间一致）+ 深度图码率压缩（量化/降分辨率，纳入"极低码率"账）；
- Qwen3-VL-4B-Instruct 升级（transformers≥4.57，结构化/差分描述）；
- 压短关键帧周期（更低生成分辨率、TensorRT、更激进蒸馏）——核心 KPI；
- 双机实时演示；N（关键帧周期）按运动幅度自适应。

## 5. 最终规划（≥8 月）

准实时遥控替代：

- 关键帧周期压到目标 KPI（多卡/更激进蒸馏）；
- 相机实时流端到端闭环；
- 大运动/遮挡/突现物体的安全鲁棒性（突现物体插帧无法"无中生有"，靠压短关键帧周期 + 动态降 N 处理）；
- 生产化部署与演示材料。

## 6. 风险与兜底

1. **保底红线（最高）**：D3（6/24）保底闭环必达。若 D1–D3 出问题，D4–D6 目标版**立即让位**，3 天余量用于稳保底——合同永远优先。
2. **GPU 是激进路线的真瓶颈**：单卡 RTX 5090，工作站下周才到。klein-9B（9B）+ Qwen3 文本编码器（8B）即便 fp8，24GB 可能紧（H2 的显存 PoC 点）。**关键缓冲**：保底版用现有 Z-Image，**不依赖 klein**，即便 klein 在 24GB 跑不动要等工作站，6/30 保底版照样达标；目标版 PoC 卡显存则部分推到工作站到位后。
3. **klein H1 翻车**：自动切 Qwen-Image-Edit-2511 + InstantX ControlNet（慢一个量级，关键帧周期账变差），但保底版不受影响。回退判据见技术方案 §6（H1：边缘 IoU>0.4）。
4. **单卡并行约束**：6 天编排的"‖ 并行"在单卡上是逻辑并行（错峰/小分辨率），非全程物理并行；工作站到位后并行度提升。

## 7. 验收口径

- **M1 保底版（6/24，合同硬目标）**：输入一段真实行车视频（8–16s）→ 输出同长度生成视频；evaluation 出逐帧+整段 PSNR/SSIM/LPIPS/CLIP；单机与双机(relay)各演示一次；记录端到端耗时（允许慢）。**通过 = 闭环跑通 + 有量化指标 + 可演示。**
- **目标版 PoC（6/27，增量、不达标不影响合同）**：关键帧+插帧+超分端到端出平滑流；测得有效输出帧率、关键帧周期、帧间一致性、端到端延迟。
- **各 PoC 通过/回退判据**：见技术方案 §6 风险清单（H1/H2/H3/插帧实时性/深度码率/Qwen 升级）。

## 8. 后续流程（重要）

**本轮到更新 ROADMAP 为止，不进 writing-plans。** 原因：6 天激进计划中 PoC 结果（尤其 H1 主线裁决、H2 速度）会直接改变后续任务的形态，现在一次性 plan 死会很快失效。改为：

- **每日任务开始时**，基于前一日 PoC/实测结果，对当天主线任务重新 brainstorm → 出当天 spec → 直接实施（小任务可跳过正式 plan）；
- **每周向负责人同步**进展 + 风险 + 本周计划；
- 本设计文档与更新后的 ROADMAP 作为每日 brainstorm 的稳定锚点（目标、边界、验收、回退不变）。
