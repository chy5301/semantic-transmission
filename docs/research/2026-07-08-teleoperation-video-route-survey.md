## 1. 调研需求锚定(讨论关注点)

本次调研在开工前经过多轮讨论逐步收窄,最终冻结的需求边界如下。**这是评判所有候选方案的标尺。**

### 1.1 应用场景
- **视频内容**:车辆/机器人的行驶/行动视频(第一/第三人称的运动画面,全场景运动)。
- **用途**:通信受限条件下**遥控(teleoperation)机器人/车辆**。
- **点名关注的模型**:NVIDIA Cosmos、Wan2.2(含社区整合版 `wan2.2-rapid-mega-aio-v12`)、Kairos3.1-4B-robot-480P、moworld(待开源);以及 DCVC、DeepJSCC、VideoMAE 等。

### 1.2 质量需求(与常规视频压缩不同)
- **需要正确**:场景与物体的**位置关系、轮廓、种类、大致外观**。
- **可牺牲**:**纹理还原度**要求不高。
- 推论:此需求指向**结构化语义表征**(分割/深度/布局/边缘),而非自由文本 caption——文本擅长"是什么",不擅长精确编码"在哪、什么形状、什么空间关系"。

### 1.3 硬约束
| 维度 | 约束 |
|---|---|
| **带宽** | 有效带宽 ≤ 1Mbps |
| **延迟** | 秒级可接受,**小时级出局**;项目指标未硬性要求延迟,但现方案"10s 视频→小时级"完全不可用。推理延迟/吞吐是**硬评分线**。 |
| **延迟测量口径** | 以**双机双卡部署**为准——接收端**独占一整张 GPU(24GB 级)** 估算重建 ~10s 片段耗时。 |
| **部署/算力** | 开发测试单卡 24GB 可跑即可;实际部署双机双卡(收发分离,各占一卡),接收端生成模型独享整卡。 |
| **训练成本** | 优先"拿来即用"的**开源预训练模型/框架**(少训练/免训练),**不锁定 Diffusers 框架**。 |

> ⚠️ 背景校正:严格 tele-driving 学界文献假设 **175ms 端到端硬预算**(紧控制环)。但本项目延迟是**秒级可接受**——比学界基线宽松得多,把很多"太慢而被否"的生成式方案重新拉回可选范围。

### 1.4 现有路线的疑虑(调研动机)
现方案:`视频→图→文本(VLM caption)→图(逐帧扩散生成)→视频`(Diffusers + Z-Image-Turbo + ControlNet Canny + Qwen2.5-VL)。
- 还原质量差;**加入原始帧参考后帧间闪烁严重**;10s 视频重建延迟**小时级**,完全不可用。
- 质疑"**文本瓶颈 + 逐帧独立生成**"路线是否应整体更换。

### 1.5 冻结的 5 个调研角度
1. **遥驾驶/遥操作专用低码率视频**:tele-driving 压缩、ROI/任务感知编码、面向驾驶的语义通信/V2X。
2. **视频生成模型/世界模型当接收端解码器 + 延迟现实性**:Cosmos/Wan2.2/CogVideoX/LTX/Hunyuan/SVD + 世界模型 Kairos/moworld;重点查免训练条件重建、单卡重建 ~10s 片段实际耗时、能否帧外推补偿延迟。
3. **结构化语义瓶颈**:分割/深度/布局/边缘 → 生成式解码。
4. **神经编解码 & 语义通信 + 传统编解码诚实基线**:DCVC 系列/DeepJSCC/VideoMAE;对比 HEVC/VVC/AV1 在 ≤1Mbps。
5. **DLSS 式接收端增强层**:超分 + 插帧/帧外推(区分:插帧降带宽增延迟 vs 外推降延迟)。

> GFVC(人脸生成压缩)仅了解不深挖——只对人脸有效,与车辆/机器人场景不符。

## 2. 核心结论

**现有路线(VLM caption 文本瓶颈 + 逐帧独立扩散)在证据下几乎是所有可选路线里最差的组合**——它同时踩中三个坑,而每一个成熟替代方案至少避开其中一个:

| 现有痛点 | 根因 | 证据支持的解法 |
|---|---|---|
| 还原质量差(位置/轮廓不准) | 文本瓶颈丢几何信息 | 改用**结构化条件**(深度/分割/边缘)——ControlVideo/ConditionVideo/LTX 原生支持,训练-free |
| 加参考帧后闪烁严重 | 逐帧独立采样,无帧间约束 | 改用**视频原生模型**(自带跨帧注意力)——ControlVideo sBiST-Attn 报 99% 帧一致性 |
| 10s 视频小时级延迟 | SD 逐帧 + 无优化 | 换**秒级/分钟级**解码器(LTX-Video、DCVC-RT) |

**"闪烁能否救"的判断:不抱期望是对的。** 逐帧独立生成是结构性闪烁,补丁救不动;正解是换视频原生解码器或时序一致超分(Upscale-A-Video)。**建议不在旧路线上继续投入,直接切换。**

### 增强层(与主线正交,可叠加任意路线)
- **超分 SR**:Real-ESRGAN / BasicVSR++ / VRT / **Upscale-A-Video(时序一致,直接治闪烁)** / Video2X(打包 SR+RIFE,但 AGPLv3 且离线批处理)——都预训练、即用。**低风险高收益,建议无脑纳入。**
- **插帧 vs 外推(关键区分,已证实)**:
  - RIFE / FILM / DLSS3 = **插帧**,需未来帧 → **增延迟**(2080Ti 上 720p 2× 可 30+fps,MIT)。
  - STSS / GFFE = **帧外推**,预测未来帧 → **降延迟**,但更难(disocclusion)且**是游戏渲染域**(需 G-buffer 或针对渲染内容),**不能直接用于摄像头视频**。
  - 结论:想降带宽→插帧可用(本项目延迟宽松);想降延迟的帧外推**对相机视频目前不成熟**。

## 4. 三条推荐路线(按研判排序)

### 🥇 路线 A:传统编解码 + ROI/任务感知(+ 可选 SR)——务必先做的基线
证据高度一致:遥操作视频的**成熟工程解法根本不是生成式**,而是"低延迟自适应编码 + 任务感知码率分配"。TUM Hofbauer 博士论文、TAMVA(MIT/ROS/CARLA)、三级 ROI-HEVC、TELECARLA 全部指向这条。它**恰好匹配本项目质量需求**(优先交通标志/物体轮廓/类别,牺牲纹理),实时、开源、当天能搭。ROI 省码率约 19.8%–40%,交通标志 PSNR +5.5dB。
**它可能"不够 AI 不够酷",但很可能是 ≤1Mbps 下性价比最高的答案。至少必须作为诚实基线,否则任何生成式方案都无法证明自己有净收益。**

### 🥈 路线 B:结构化条件 → LTX-Video 生成式解码(若坚持走语义/生成路线)
**丢掉文本瓶颈**,发送端传"关键帧 + 深度/分割/Canny",接收端用 **LTX-Video**(原生 Depth/Pose/Canny 控制 + 多关键帧条件)重建。这一条**同时修好三个痛点**:结构条件保几何、视频模型保时序、秒级不再小时级。LTX <2B、能塞进 24G,是当前"快 + 结构可控 + 即用"最平衡的生成式解码器。
**核心待验证**:H100 上的秒级数字**不能直接搬到 24G 卡**,必须在双卡环境实测 ~10s 片段真实耗时——这是决定 B 生死的唯一实验。

### 🥉 路线 C(观望):世界模型
Cosmos 的 Video2World(关键帧+文本预测,支持相机位姿/动作条件)概念上**最贴合车辆/机器人**,但:论文**零延迟数据**、10000×H100 训练的重模型、且"世界模型免训练当解码器"证据不足(surgical 世界模型那篇是**微调了 ~2000 clip** 才可用,非 zero-shot)。moworld 待开源可持续跟踪。**先别投入,等延迟实测或更轻的开源世界模型出现。**

## 5. 下一步实测清单(可落地)

1. **建基线**:TELECARLA/TAMVA 跑通 ROI-HEVC/H.265,测 ≤1Mbps 下的主观可用性(能否看清标志/车/人的位置轮廓)。
2. **测神经编解码**:DCVC-RT(MIT,即用),看它在驾驶素材(大背景运动)上是否稳定——证据显示 DCVC-DC/FM 在大背景运动上不稳,DCVC-**RT** 要单独验。
3. **测生成式解码**:LTX-Video + Depth/Canny 控制,在**接收端单卡**实测 10s 片段延迟与几何保真,和路线 A 基线对比有无净收益。
4. **叠加层**:任意主线上叠 Real-ESRGAN 或 Upscale-A-Video(治闪烁),验证质量增益。

## 6. 关键发现细目(带引用)

### 6.1 遥操作专用低码率方案(角度①)
- **TAMVA**:开源 MIT、ROS + CARLA,交通感知多视角自适应,较无预处理的多路复用**总码率降 40%**,同时保住最重要视角质量。
- **三级 ROI-HEVC**:用语义分割把 CTU 分 background/weak-ROI/strong-ROI(交通标志/灯优先),交通标志感知质量**+5.5dB** vs 二值前景/背景划分。
- **ROI 掩码**:后向摄像头 ROI 掩码 BD-rate **省 ≥19.8%** vs 整幅传输;经真人操作员评测显著提升任务表现、降低负荷。
- **TUM Hofbauer 博士论文(2022)**:把遥驾驶视频定义为"带宽受限移动网络下的低延迟自适应流",核心是编解码码率控制适配(帧率/尺寸/码率/质量),**非生成式重建**——单帧级码率控制粒度(低延迟要求)。开源 **TELECARLA**(CARLA 扩展,提供码率控制接口)。
- **175ms 硬预算**:安全遥驾驶要求端到端 ≤175ms,glass-to-glass ~100ms(严格控制环场景;本项目已放宽)。

### 6.2 视频/世界模型当解码器 + 延迟(角度②)
- **LTX-Video**:H100 上 **2 秒**生成 5s/24fps/768×512(121 帧);蒸馏 13B **~10s/HD**,fp8 **实时**;<2B 参数,VAE 1:192 压缩;原生 I2V + 多关键帧条件 + Depth/Pose/Canny 控制(IC-LoRA);OpenRAIL-M;蒸馏 LoRA 仅需 1GB 显存。**当前最平衡的生成式解码器。**
- **Wan 2.2 TI2V-5B**:24GB 单卡可跑,Apache-2.0,免训练;RTX4090 **~9min/5s 片段**(→10s 约 18min,分钟级);可叠 Real-ESRGAN 到 1440p/4K。A14B MoE 需 FP8/Q8 + offload。
- **NVIDIA Cosmos**:开权重(NVIDIA Open Model License),7B/14B diffusion + 4B/12B 自回归;Video2World=当前观测(输入视频)+文本 预测未来视频;支持相机位姿/动作条件(机器人向)。**但论文无端到端推理延迟数据**,tokenizer 34.8ms/帧@720p;10000×H100 训练三个月的重模型。→ ~10s 片段单卡耗时**必须实测**。
- **世界模型现实性**:surgical world model 论文用 LTX-Video/HunyuanVideo **微调**(LoRA + 全量,~2000 标注 clip)才可用,**非 zero-shot**——印证"世界模型免训练当解码器"目前基本不成立。

### 6.3 结构化语义瓶颈(角度③)
- **ControlVideo**:训练-free,SD1.5 + ControlNet 无微调,支持 Canny/深度/人体姿态;sBiST-Attn(稀疏双向时空注意力)报 **99.02% 帧一致性**;MIT;512²/24 帧 ~90s(sBiST-Attn),旧 SD1.5 基线。
- **ConditionVideo**:训练-free,SD1.5 + ControlNet,支持 pose/depth/canny/segmentation;直接支撑"结构表征优于自由文本"的论点。
- 现代快速版即 **LTX-Video 原生 Depth/Pose/Canny 控制**。

### 6.4 神经编解码 & 语义通信 vs 传统基线(角度④)
- **DCVC-RT**:**首个 100+fps 1080p 神经编解码**(A100 上编/解 125.2/112.8 fps),比 **H.266/VVC 省 21% 码率**;MIT + 预训练权重;支持任意分辨率。**直接推翻"神经/生成式必然慢"的假设。**
- **诚实基线的反面证据**:低延迟配置下,**最强传统编解码 JVET ECM 比 DCVC-FM 好 11.0% BD-rate**——学习型编解码在延迟敏感场景**不一定赢**传统。且 **DCVC-DC/DCVC-FM 在大背景运动上不稳定**(正是驾驶素材特征)。
- **传统编解码效率序**:VVC > AV1 > HEVC > AVC。VVC 比 AVC 省 ~59%(FHD)~93%(8K);但**编码复杂度极高**(比 AV1 慢 27–174×,比 HEVC 约 10–20×),发送端实时性受限。**注意:VVC/AV1 增益主要在高分辨率,≤1Mbps/480p 低分辨率档位增益打折**(测试集 1–15Mbps FHD/UHD,未直接覆盖目标档位)。
- **DeepJSCC/语义通信**:增益小(-33% 符号 / +0.56dB PSNR),需训练,无 turnkey 预训练权重(OpenSemanticComm 仅为索引);多针对无线图像传输,非驾驶/遥操作。

### 6.5 DLSS 式增强层(角度⑤)
- **超分**:Real-ESRGAN、BasicVSR++、VRT、**Upscale-A-Video(CVPR2024,时序一致,直接治闪烁)**、mmagic 打包;Video2X 打包 SR+RIFE(AGPLv3,离线批处理)。
- **插帧(增延迟)**:RIFE(30+fps 720p 2× on 2080Ti,MIT,预训练);Video2X/REAL-Video-Enhancer 打包。
- **帧外推(降延迟,但受限)**:STSS(统一超分+外推,~4ms,省 75% 时间 vs 两阶段 17ms)、GFFE(免 G-buffer 外推);但**均为游戏渲染域**,disocclusion 是固有难点,**不直接适用于相机视频**。
