# 项目综合评估设计（2026-06）

> 状态：已确认执行
> 产出物：`docs/research/2026-06-reevaluation.md`

## 背景与动机

unify-config-and-loader workflow（PR #42）合入后，配置体系与模型加载抽象已统一，
open issue 从 26 个降至 16 个。但项目存在结构性问题：

- 近两个月的工作全部集中在阶段二/四的工程化（脱离 ComfyUI、CLI/GUI、配置统一），
  ROADMAP 阶段三的核心研究（视频帧序列、连续帧一致性、条件升级、模型对比）尚未启动；
- 核心目标是"视频语义传输"，当前 pipeline 仍以单图为主；
- #41 提出的换模型需求（FLUX.2-klein-9B）与连续帧解码思路（上一帧 + 边缘图双参考）
  超出了 2026-03 选型报告的时效范围；
- 团队对"全面重构 vs 继续迭代"缺乏判断依据。

## 评估要回答的问题

1. **技术路线层**
   - 接收端文生图模型再选型：Z-Image-Turbo vs FLUX.2-klein-9B vs 2026 年中其他候选。
     硬约束：图像编辑 / 参考生成能力（连续帧解码的关键前提）+ 24GB 显存主档，
     另给 64GB+ 档备选升级路径。
   - 连续帧解码策略：「上一帧 + 边缘图」双参考的可行实现路径与帧间一致性方案。
   - 视频生成路线：现在引入视频生成模型，还是图像模型逐帧 + 一致性约束。
   - 发送端 VLM 是否需要升级。
2. **架构层**
   - 现有抽象（ModelLoader / Receiver / pipeline / relay）能否支撑
     「换模型、上一帧参考、视频帧序列」三个新需求，最小改动面是什么。
   - #40 各子问题（VRAM 临界、生命周期、双机演示、通信架构）"挡路 / 不挡路"判定。
3. **规划层**
   - 剩余 16 个 open issue 按「挡核心目标 / 纯工程债 / 可关闭」triage。
   - 修订版 ROADMAP 草案与下一个实施 workflow 的启动建议。

## 评估方法

动态多 agent workflow，两阶段：

**调研阶段（6 条线并行）**

| 线 | 内容 | 信息源 |
|---|------|--------|
| 1 | 接收端文生图模型生态（FLUX.2-klein-9B 深查 + 横向扫描，两档输出） | Web |
| 2 | 连续帧解码策略（双参考实现路径 + 一致性方案 + 漂移问题） | Web |
| 3 | 视频生成路线（Wan / HunyuanVideo / LTX 等现状 + 路线对比） | Web |
| 4 | 发送端 VLM 候选 + 语义通信学界 2025-2026 进展 | Web |
| 5 | 代码库架构审计（三个新需求支撑度 + #40 挡路判定） | 本地代码 |
| 6 | issue 全量 triage（16 个 open issue 分类） | 本地 + gh |

**综合阶段**：synthesis 汇总六线结果 → 评估报告。

## 决策点（保留给负责人）

报告完成后人工评审，决定：是否换模型、下一个 workflow 的 spec 方向、
ROADMAP 是否按草案修订。之后按项目惯例用 structured-workflow 启动实施
（一个 workflow 一个分支/PR，启动前重新 brainstorm spec）。

## 约束记录

- 硬件：现阶段双端均按 RTX 5090 24GB 评估；未来可能购置 64GB+ 设备（未定），
  选型需给出可平滑升级的备选。
- 时间：无硬演示节点（约 5-6 个月窗口），评估导向是"系统性解决推进卡顿"，
  平衡快速出效果与避免返工。
- 网络：模型下载依赖 HF 镜像站（hf-mirror）/ ModelScope，选型需考虑国内可获取性。
