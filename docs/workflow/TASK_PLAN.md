# 任务计划

## 总体策略

**按主题分块调研**：将调研工作按论文综述、开源项目、模型能力三个维度分块推进，各块相对独立，可并行执行，最后汇总为统一的选型报告。

选择理由：
1. 三个维度的信息源不同（论文库、GitHub、模型官方文档），分块效率更高
2. 每块独立产出文档，便于增量更新和团队共享
3. 汇总阶段可交叉对比，形成更有说服力的选型建议

## 阶段里程碑

| 阶段 | 名称 | 退出标准 |
|------|------|----------|
| Phase 0 | 准备 | 调研框架和目录结构就绪，调研范围明确，辅助工具就绪 |
| Phase 1 | 论文与项目调研 | 核心论文和开源项目收集完成，各方案优劣已整理 |
| Phase 2 | 模型能力调研 | 关键模型（理解/生成/控制）能力评估完成 |
| Phase 3 | 汇总与选型报告 | 调研报告完成，技术选型建议明确 |

### Phase 0: 准备

#### [G-00] 搜索评估-调研辅助Skill

- **阶段**: Phase 0 - 准备
- **依赖**: 无
- **目标**: 搜索并评估可加速调研工作流的 Claude Code skill（如网页内容抓取、论文搜索、文档生成等），确认安全性后安装到项目级别
- **背景信息**: Claude Code 支持通过 skill 扩展能力。本工作流的调研任务涉及大量网页搜索、论文阅读、信息整理等工作，合适的 skill 可以显著提升效率。需要在 skill 市场中搜索相关工具，评估其功能匹配度、安全性（是否有恶意代码、权限是否合理）和社区信誉（Star 数、安装量、维护者），然后将通过评估的 skill 安装到项目级别供后续任务使用。
- **涉及文件**:
  - `docs/research/skill-evaluation.md`（新建，skill 评估记录）
- **具体步骤**:
  1. 使用 `npx skills find` 搜索与调研相关的 skill（关键词：research、web、fetch、paper、arxiv、document、summary 等）
  2. 筛选可能有用的 skill 候选清单
  3. 逐个评估候选 skill：
     - 功能匹配度：是否对论文搜索、网页内容提取、文档整理有帮助
     - 安全性：检查 skill 来源仓库的代码，确认无恶意行为（如数据外传、敏感信息读取）
     - 社区信誉：Star 数、安装量、维护者身份、最近更新时间
  4. 将通过评估的 skill 安装到项目级别：`npx skills add "<owner/repo>" --skill "<name>"`
  5. 编写评估记录文档，包含评估过的 skill 清单、评估结论和安装结果
- **验收标准**:
  - [ ] 至少搜索了 5 个相关关键词
  - [ ] 每个候选 skill 有安全性和有效性评估记录
  - [ ] 通过评估的 skill 已安装到项目
  - [ ] 评估记录文档已完成
- **自测方法**: 运行 `npx skills list` 确认 skill 已安装；检查评估记录文档的完整性
- **回滚方案**: 使用 `npx skills remove` 卸载已安装的 skill，删除评估文档
- **预估工作量**: M (约 1 小时)

### Phase 1: 论文与项目调研

#### [G-03] 调研-语义通信核心论文

- **阶段**: Phase 1 - 论文与项目调研
- **依赖**: G-01
- **目标**: 收集并整理 ≥5 篇语义传输/语义通信领域的核心论文，提取关键方法、实验指标和结论
- **背景信息**: 语义传输是一个新兴研究方向，核心思路是"传输语义而非像素"。已知的代表性工作包括：GVSC（生成式视频语义通信，多模态融合+扩散模型）、GVC（TeleAI，0.01% 压缩率）、GSC（超低码率语义编码）、扩散辅助极端视频压缩、M3E-VSC（多模态互增强）。需要系统阅读这些论文，提取对本项目有指导意义的信息。
- **涉及文件**:
  - `docs/research/papers/semantic-communication-survey.md`（新建）
- **具体步骤**:
  1. 在 arXiv、Google Scholar 搜索语义传输/视频语义通信相关论文（2024-2026）
  2. 重点阅读 GVSC、GVC、GSC、M3E-VSC 等已知工作
  3. 扩展搜索相关引用和被引论文
  4. 按统一模板整理每篇论文：方法概述、系统架构、压缩率、质量指标（PSNR/SSIM/LPIPS/CLIP Score）、优势与局限
  5. 总结各方案的共性和差异，提炼对本项目的启示
- **验收标准**:
  - [ ] 至少 5 篇核心论文已整理
  - [ ] 每篇论文包含方法、指标、优劣势的结构化摘要
  - [ ] 论文间的方法对比表已完成
  - [ ] 对本项目的启示和建议已总结
- **自测方法**: 检查论文数量和对比表的完整性
- **回滚方案**: 删除 `docs/research/papers/semantic-communication-survey.md`
- **预估工作量**: L (约 2.5 小时)

### Phase 2: 模型能力调研

#### [G-05] 调研-视觉理解模型

- **阶段**: Phase 2 - 模型能力调研
- **依赖**: G-03
- **目标**: 对比评估适用于发送端（视频/图像→文本描述）的多模态大模型的能力
- **背景信息**: 语义传输的发送端需要将视频帧转换为准确的文本描述。这要求模型能捕捉场景布局、物体关系、颜色、纹理等细节。候选模型包括 Qwen-VL 系列、InternVL、LLaVA、GPT-4V 等。评估维度包括：描述精度、推理速度、部署难度、是否支持视频输入等。现有 ComfyUI 工作流使用 qwen_3_4b 作为 text encoder，但发送端的图像描述生成需要更强的视觉理解能力。
- **涉及文件**:
  - `docs/research/models/visual-understanding-models.md`（新建）
- **具体步骤**:
  1. 列出候选模型清单（Qwen-VL-Max/Plus、InternVL2.5、LLaVA-OneVision 等）
  2. 收集各模型的官方 benchmark 数据（图像描述、视频理解等任务）
  3. 对比各模型的参数量、推理速度、部署要求（显存、框架）
  4. 评估各模型对"结构化场景描述"的能力（能否按模板生成分层描述）
  5. 整理对比表，给出发送端模型的初步推荐
- **验收标准**:
  - [ ] 至少 3 个候选模型已对比
  - [ ] 对比维度包含：描述精度、速度、部署要求、视频支持
  - [ ] 模型对比表已完成
  - [ ] 有初步推荐和理由
- **自测方法**: 检查模型对比表的完整性和推荐的合理性
- **回滚方案**: 删除 `docs/research/models/visual-understanding-models.md`
- **预估工作量**: M (约 1.5 小时)

#### [G-07] 调研-条件控制与ControlNet方案

- **阶段**: Phase 2 - 模型能力调研
- **依赖**: G-02, G-06
- **目标**: 对比不同条件控制方式（Canny、深度图、语义分割等）在语义传输场景下的适用性
- **背景信息**: 语义传输中仅靠文本描述还原图像精度有限，需要传输轻量的结构化条件信息来提高还原准确度。条件类型影响两个方面：1）传输数据量（不同条件的压缩效率不同）；2）还原质量（不同条件保留的信息维度不同）。现有工作流使用 Canny 边缘（低阈值 0.15，高阈值 0.35）+ Z-Image-Turbo-Fun-Controlnet-Union。需要评估其他条件类型是否更适合语义传输场景。
- **涉及文件**:
  - `docs/research/models/controlnet-conditions.md`（新建）
- **具体步骤**:
  1. 列出主流 ControlNet 条件类型：Canny、深度图、法线图、语义分割、姿态、线稿等
  2. 分析各条件类型的信息保留特点（保留什么、丢失什么）
  3. 估算各条件类型的数据量（传输成本）
  4. 收集论文中关于不同条件类型对还原质量影响的实验数据
  5. 评估 Union ControlNet（多条件融合）的可行性和效果
  6. 给出语义传输场景下的条件类型推荐
- **验收标准**:
  - [ ] 至少 4 种条件类型已对比
  - [ ] 信息保留特点和数据量估算已完成
  - [ ] 有基于论文数据的质量对比
  - [ ] 条件类型推荐有明确理由
- **自测方法**: 检查条件类型对比表和推荐的合理性
- **回滚方案**: 删除 `docs/research/models/controlnet-conditions.md`
- **预估工作量**: M (约 1 小时)
